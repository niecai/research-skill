# Top-Down Debugging

## 何时触发

满足任一条就触发：

- 用户指出“你在打转”“一直试错”“越改越乱”
- 你已经连续做了 2 次同层 probe，信号仍然不收敛
- 症状越来越多，但控制层级还没画出来
- 你开始反复改 `slot / route / trigger / 热键 / mock / probe / 配置项 / 局部参数`
- 已经存在可信 handoff、working 样本、设计文档、运行日志或别人的层级分析，但你还没把它提升成主模型
- 问题还没形成结构，你却已经开始实现或打补丁

触发后先对用户说一句：

`我在用 top-down-debugging，先重画层级，再决定下一轮实验。`

## 目标

先回答这句话：

`working 比 failing，多了哪个控制点，少了哪个错误控制点，或者哪一层根本走错了。`

不要先回答：

- “再试一个参数”
- “再换一个入口”
- “再补一个临时 patch”

## 核心规则

### 1. 先接管可信层级

如果已经有可信来源把问题层级写清楚了，先把它当主模型，不要把它当背景阅读。

可信来源包括：

- working 样本、baseline、线上正常版本
- 可靠 handoff 或复盘
- 已验证的 pipeline 文档
- 可复现的日志、抓包、截图、录屏、指标
- 已经跑通的工具链说明

你必须先把那套层级改写成当前任务的检查表，再决定实验。

### 2. 强制四方对照

开始前至少明确 4 个对象：

1. `working`：已知正确或更接近正确的版本
2. `failing`：当前失败版本
3. `reference`：源输入、协议、规格、源数据、目标行为或外部标准
4. `layer model`：当前采用的控制层级图

缺任何一个，都先补，不要继续局部修补。

### 3. 先找层，再找点，再找值

调试顺序固定为：

1. 哪一层出了问题
2. 这一层缺了哪个控制点
3. 这个控制点里的哪个值不对

不要跳过前两步直接调值。

### 4. 每轮实验只回答一个问题

好实验必须是单变量的，并且写清：

- 这轮要证伪什么
- 改的是哪一层
- 不碰哪几层
- 成功和失败分别意味着什么

如果一轮实验同时改了两层以上，这轮结果默认不干净。

### 5. 两次负信号后强制升层

同一层连续两次负信号后，必须停止在该层硬试，回到上一层重画控制图。

不要出现：

- “再试一个哈希”
- “再换一个 payload”
- “再补一个 route”
- “再加一个开关”

除非你先说明：上一层为何已经被证伪。

### 6. 已知无效家族直接禁用

如果可信层级已经说明某个家族在当前上下文里是无效的、静音的、不会命中的、不会加载的、不会被路由到的，就不要继续在这个家族内部微调。

正确动作：

- 把“家族选择错误”记为当前结论
- 升层到“正确家族是什么”
- 下一轮实验只验证新的家族，不再回头修旧家族

## 通用控制层级模板

不是每个问题都要用全套，但至少要明确你当前在第几层：

1. **目标行为层**
   - 用户真正想看到什么
2. **运行时表现层**
   - 现在实际发生了什么
3. **路由/触发层**
   - 是谁把这条链路启动起来的
4. **主资产层**
   - 核心代码、动画、数据、模型、配置主体
5. **sidecar / 元数据层**
   - 附属事件、音效、判定、时序、映射表、缓存、索引、标签、权限
6. **运行时上下文层**
   - 房间、角色、阵营、场景、网络、权限、平台、资源包、环境变量
7. **外部依赖层**
   - 工具链、打包器、格式转换器、服务、驱动、第三方库

## 每层必须问的问题

对每一层都问：

- `working` 有什么而 `failing` 没有
- 是“值不对”，还是“根本没传到这层”
- 是“主资产有问题”，还是“sidecar 残留”
- 是“文件就错了”，还是“运行时上下文改写了结果”
- 这一层的结论来自证据，还是来自猜测

## 最小差异表

继续修改前，先产出最小差异表，至少包含：

- `working / failing / reference / layer model`
- 当前比较的是哪一层
- 本层最关键的 1-5 个控制点
- 发现的差异：缺失、错位、残留、越层、上下文依赖
- 下一轮实验要证伪的问题

## 常见打转模式

### 模式 A：把可信层级当背景材料

症状：

- 已有文档明说问题在某层，你还在下一层微调
- working 样本已经证明某条路不成立，你还在重复局部 probe

处理：

- 先把文档里的层级提升成当前主模型
- 用主模型重新标记当前实验所在层

### 模式 B：把 sidecar 问题误判成主资产问题

症状：

- 主体代码、动作、数据或流程是对的
- 但音效、判定、时序、缓存、权限、附件、特效不对

处理：

- 冻结主体
- 只查 sidecar 与上下文层

### 模式 C：把上下文问题误判成文件问题

症状：

- 房间内外、角色不同、网络状态不同、权限不同，结果就变
- 同文件在不同运行时上下文里表现不一样

处理：

- 把“上下文”单独拉成一层
- 不要继续只改文件内容

### 模式 D：同层反复探针

症状：

- 换 payload、哈希、阈值、入口、时序，结果只是换一种错法

处理：

- 两次负信号后强制升层
- 回答“为什么这一层值得继续调”

### 模式 E：研究问题直接开始实现

症状：

- 还没弄清控制图就开始写代码、改配置、跑实验
- 结果产出越来越多，但问题空间没有缩小

处理：

- 先补 `working / failing / reference / layer model`
- 先做差异表，再决定实现顺序

## 代理自约束

如果你开始反复说这些话，就说明该触发本 skill：

- “再试一次”
- “可能是这个参数”
- “先改这里看看”
- “先让用户再测一下”
- “我大概知道方向了，先做一版”

这通常说明你已经回到局部试错，而不是在缩小问题空间。

## 会后复盘

如果这次已经打转，结束前补一段：

- 我打转在哪一层
- 为什么会打转
- 哪条可信层级没有被及时接管
- 下次在第几步就该停下来

把这段复盘写回 skill 或项目文档，而不是只留在对话里。
